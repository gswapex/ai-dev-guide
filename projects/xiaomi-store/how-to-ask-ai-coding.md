# 如何向 AI 提问，真正学会做项目

> AI 提问指南 | 面向实战开发学习者
> 小米之家门店管理系统 — Spring Boot + React 实战陪跑版

---

## 一、为什么你需要这份指南

很多同学在遇到问题时，第一反应是直接问 AI："帮我写这个代码"、"这里为什么报错"。这样问 AI 当然能拿到一个答案——但你真的学会了吗？

问题出在"问法"上。太粗的问题只能得到泛泛的回答；太笼统的问题让 AI 无从发力。本指南的目标只有一个：

> **教你把"我不会"转化为"我能问清楚、学明白、做出来"。**

---

## 二、六类高价值提问方向

按照开发过程中最常遇到的困惑，整理成以下六类提问方向，每类都附有差用问法和好用问法的对比。

---

### 2.1 代码思路类 — "为什么要这样设计？"

**适用场景：** 看到一段代码或一个方案，但不理解背后的设计意图。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 帮我写一个创建订单的接口 |
| ✅ 好问法 1 | 创建订单时为什么要把商品价格存入 OrderItem，而不是直接关联 Product 的价格字段？这样设计的好处是什么？ |
| ✅ 好问法 2 | 我在 OrderServiceImpl 里看到 @Transactional 注解，这里加事务是为了解决什么问题？如果不加会出现什么情况？ |
| ✅ 好问法 3 | 库存管理为什么要单独建 STOCK_LOG 表记录每次变动，而不是直接在 PRODUCT_INFO 里加一个字段？ |

**提问框架：**

1. 为什么要用 [X] 而不用 [Y]？
2. 这样设计是为了解决什么问题？
3. 如果不这样做会出现什么情况？
4. 什么场景下这种设计会失效或出问题？

---

### 2.2 代码执行逻辑类 — "这段代码怎么跑起来的？"

**适用场景：** 代码能运行但不清楚每一步在干什么，或者搞不懂数据是怎么流转的。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 这段代码什么意思 |
| ✅ 好问法 1 | 用户调用 POST /v1/auth/login 之后，请求从 Controller 到 Service 再到 Repository，每一层分别做了什么？数据在哪个环节被加密验证的？ |
| ✅ 好问法 2 | JWT 拦截器是在什么时机被触发的？它读取 Token 之后把用户信息存在哪里，后续 Controller 如何取出来使用？ |
| ✅ 好问法 3 | 前端 api/request.js 里的响应拦截器，具体在什么情况下会触发 401 跳转？整个执行顺序是怎样的？ |

**提问框架：**

1. 从 [入口] 到 [出口]，中间经历了哪几个步骤？
2. 这个数据是从哪里来的，经过哪些处理，最终存/返回到哪里？
3. 这段逻辑第一步是什么，第二步是什么？有没有条件分支？
4. 能不能用一个具体的例子，带着数据走一遍这个流程？

---

### 2.3 操作步骤类 — "下一步我该做什么？"

**适用场景：** 知道要实现某个功能，但不知道从哪里下手，或者不清楚步骤顺序。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 帮我做订单模块 |
| ✅ 好问法 1 | 我要实现"创建订单"功能，按照后端分层架构，应该先建哪个文件，写哪些代码？给我一个完整的开发步骤顺序。 |
| ✅ 好问法 2 | 我已经完成了 Entity 和 Repository，下一步写 Service 时，createOrder 方法内部的逻辑应该按什么顺序实现？ |
| ✅ 好问法 3 | 前端订单列表页 Order/index.jsx，我应该先写哪个部分——是先写静态 UI 还是先接通接口？整体开发顺序是什么？ |

**提问框架：**

1. 实现 [功能X]，从零开始的步骤是什么？先做什么，再做什么？
2. 我已经完成了 [A]，下一步该做什么？
3. 这个功能涉及哪些文件？我应该按什么顺序创建/修改它们？
4. 如果要验证这一步是否做对了，我该怎么测试？

---

### 2.4 性能问题类 — "哪里会慢？怎么优化？"

**适用场景：** 功能实现后，想了解性能瓶颈在哪里，以及数据量变大时会出什么问题。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 这个接口怎么优化 |
| ✅ 好问法 1 | 获取商品列表接口在商品数据量达到10万条时，有哪些可能的性能瓶颈？现在的 SQL 查询是否需要加索引？ |
| ✅ 好问法 2 | 统计接口 GET /v1/reports/statistics 每次请求都做聚合查询，如果并发量大会怎样？有什么缓存方案可以优化？ |
| ✅ 好问法 3 | 库存出库接口在高并发场景（多人同时购买同一商品）下会不会出现超卖问题？如何用数据库级别解决这个问题？ |
| ✅ 好问法 4 | 前端商品列表页每次切换分页都要发请求，有没有办法做数据缓存减少不必要的请求？ |

**提问框架：**

1. 当数据量达到 [N] 时，[接口/功能] 的性能瓶颈在哪里？
2. 这里的时间复杂度/SQL 查询效率是怎样的？有没有更好的写法？
3. 在高并发场景下，[功能X] 会出现什么问题？如何保证数据一致性？
4. 哪些地方适合加缓存？加了缓存后需要注意哪些数据一致性问题？

---

### 2.5 调试排错类 — "为什么报错了？从哪里查？"

**适用场景：** 遇到报错或接口返回不符合预期，不知道从哪里开始定位问题。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 报错了怎么办（不附任何信息） |
| ✅ 好问法 1 | 调用 POST /v1/orders 时，控制台打印了 "NullPointerException at OrderServiceImpl.java:52"，这个错误通常是什么原因导致的？我应该先检查哪些地方？ |
| ✅ 好问法 2 | 前端调用接口时浏览器报了 CORS 错误，但后端已经配置了 CorsConfig，可能还有哪些原因导致跨域没有生效？ |
| ✅ 好问法 3 | 接口返回了 200 但 data 字段是 null，这种情况通常在哪个层出了问题？我应该按什么顺序排查：Controller → Service → Repository？ |
| ✅ 好问法 4 | 库存出库后，商品的 stockQuantity 没有减少，但接口返回了成功。这种"接口成功但数据没变"的情况，可能是什么原因？ |

**提问框架（附报错时必须提供的信息）：**

1. 报错信息是什么？（完整复制，不要截图描述）
2. 报错发生在哪个文件的第几行？
3. 这个操作之前我做了什么？触发条件是什么？
4. 我已经检查过哪些地方了？排除了哪些可能性？

---

### 2.6 方案比较类 — "A 和 B 哪个更适合我？"

**适用场景：** 有多种实现方式不知道选哪个，或者想了解不同方案的适用场景和取舍。

| 问法类型 | 示例（结合本项目） |
|----------|-------------------|
| ❌ 弱问法 | 用 MyBatis 还是 JPA？ |
| ✅ 好问法 1 | 这个项目用 Spring Data JPA 还是 MyBatis 更合适？考虑到是课程练习项目，各自有什么优缺点？初学者哪个更容易上手？ |
| ✅ 好问法 2 | 订单状态用整型（0/1/2）还是用枚举字符串（PENDING/PAID/CANCELLED）存储？两种方式在可读性和扩展性上有什么区别？ |
| ✅ 好问法 3 | 前端管理全局用户信息，用 React Context 还是 Zustand/Redux？对于本项目只有当前用户这一个全局状态，哪种方案更合适？ |
| ✅ 好问法 4 | 库存预警是在每次查询时实时计算，还是在入/出库时提前写入预警标记字段？两种方案的性能和实现复杂度有什么差异？ |

**提问框架：**

1. [方案A] 和 [方案B] 的核心区别是什么？
2. 在 [当前项目规模/场景] 下，哪种更合适？
3. 如果将来项目变大/需求变化，哪种方案扩展性更好？
4. 两种方案各自的学习成本和实现复杂度怎么样？

## 三、实战提问模板

每次遇到困惑，可以套用以下模板，确保 AI 能给出最有价值的回答：

### 通用提问模板

```
【背景】我在做小米之家门店管理系统的 [模块名] 功能，目前进展到 [具体位置]。
【问题】[具体描述你的困惑，参考六类提问方向]
【我已知道的】[说明你已经理解的部分，避免 AI 重复解释]
【我想得到的】[说明你希望得到什么：代码示例/步骤说明/原理解释/方案对比]
```

### 报错提问模板

```
【操作】我调用了 [接口路径]，传入了 [参数简述]。
【报错信息】[完整错误信息，包含文件名和行号]
【期望结果】我期望接口返回 [正确结果描述]
【我已排查的】我已经检查了 [A、B、C]，排除了 [D] 的可能性。
```

## 五、一句话记住这份指南

> **不要只问"帮我写"，要问"为什么这样写、怎么执行、下一步怎么做、哪里会慢、报错从哪查、A 和 B 哪个好"。**

这六类问法对应了六种思维方式：设计意图、执行过程、操作路径、性能意识、调试策略、方案权衡。每一种都是工程师必备的基本能力。
