# 需求规格说明助手

---

## 任务目标

本 Skill 用于将零散、非结构化的业务原始需求转化为标准化的需求规格说明文档。

| 能力项 | 说明 |
|---|---|
| 需求分析与提取 | 从原始材料中识别并提炼关键业务需求 |
| 结构化内容创作 | 按标准模板生成规范化需求规格说明文档 |
| 需求完整性校验 | 检查需求描述的清晰度、可验证性与完整性 |

**触发条件：** 用户提供业务需求描述、需求清单、PRD 草稿等原始材料，希望获得规范的需求规格说明。

- **依赖说明：** 无

## 操作步骤

### 标准流程

#### 第一步：分析输入材料

- 仔细阅读用户提供的业务原始需求（文本、文档、清单等）
- 识别并提取关键信息：业务背景、核心目标、目标用户、主要功能点、约束条件等
- 标注模糊或不完整的信息点，主动向用户确认或提出澄清问题

#### 第二步：生成需求规格说明文档

- 根据需求规格说明模板中的模板结构，逐项填写
- 确保所有必要字段都已完整填写（见模板中的"必要字段"标记）
- 保持内容的专业性和逻辑性，使用清晰的表达和结构化的格式

#### 第三步：验证与优化

- 确保需求描述清晰、无歧义、可验证
- 按高 / 中 / 低标注需求优先级，支持分期规划
- 聚焦核心内容，避免过度细节

---

### 可选分支

| 场景 | 处理方式 |
|---|---|
| 输入材料严重不足 | 主动提出引导性问题，帮助用户补充关键信息 |
| 需求涉及多个子系统 | 按子系统划分章节，提供清晰的模块边界 |
| 存在需求优先级 | 在功能需求中标注优先级（高 / 中 / 低） |

## 注意事项

- **严格按照模板结构**组织内容，聚焦核心需求（需求背景、需求目标、需求范围、用户画像、功能需求）
- **使用可验证语言**描述需求，如"支持"、"实现"等，避免"尽可能"、"考虑"等模糊表达
- **为每个功能需求提供**清晰的验收标准和优先级（高 / 中 / 低），便于后续测试和分期实施
- **充分挖掘隐性需求**，从原始材料中识别用户未明确表达但实际需要的功能
- **控制文档篇幅**，避免 token 超出，每个功能需求控制在 5～8 个验收标准内

## 使用示例

### 示例 1：电商订单管理系统

| 项目 | 说明 |
|---|---|
| 功能说明 | 将业务口述的订单管理需求转化为规范文档 |
| 执行方式 | 智能体自然语言处理 |
| 关键参数 | 业务背景、订单流程、用户角色、核心功能 |

**用户输入：**

> 我们需要一个订单管理系统，用户可以创建订单、支付、查看订单状态。

**智能体分析：**

提取订单创建、支付、状态查询等核心功能，补充订单取消、退款、物流跟踪等完整流程。

**输出结果：**

包含需求背景、目标范围、功能需求（订单管理、支付管理、物流管理）的完整文档。
