# Product Story — 小米之家门店管理系统

> 这份文档记录的不是"这个系统有什么功能"，
> 而是"这个系统为什么长成这样"。
> 读完它，你就能理解需求是怎么一步一步演变出来的。

---

## 第一步：发现问题

一家小米之家门店，每天要处理这些事情：

- 店员卖出一部手机，要手动记一下库存减了多少
- 顾客问某款产品还有没有货，店员要跑去仓库看
- 月底经理要统计这个月卖了多少、收了多少钱，只能翻流水账
- 会员积分靠纸质卡片记录，顾客换手机号了就找不回来了

**一句话总结：门店每天的运营数据是散的，靠人记、靠纸记，效率低、容易出错。**

> 💬 思考一下：你觉得这些问题里，哪一个最影响门店的日常运营？

---

## 第二步：确定用户

这个系统不是只给一种人用的，门店里有几类人，每个人的需求不一样：

| 角色 | 每天在做什么 | 最需要什么 |
|------|------------|-----------|
| 店员 | 卖货、开单、登记会员 | 快速查库存、快速录订单 |
| 门店经理 | 管商品、审库存、看业绩 | 实时看数据、管商品价格 |
| 财务人员 | 统计销售额、出报表 | 自动汇总、导出报表 |
| 系统管理员 | 管账号、管权限 | 配置权限、维护系统 |

**关键发现：不同角色看的数据不一样，能操作的功能也不一样。**
所以这个系统从一开始就需要"角色权限"这个设计——不是为了复杂而复杂，而是因为现实就是这样。

> 💬 思考一下：如果不做权限控制，店员能看到经理的销售报表，会有什么问题？

---

## 第三步：功能是怎么定出来的

知道了谁在用、遇到了什么问题，功能自然就推出来了：

**店员遇到的问题 → 对应的功能**

```
"卖货要手动记库存" → 订单管理 + 库存自动扣减
"顾客问有没有货"  → 库存查询 + 库存预警
"会员积分靠纸记"  → 会员管理 + 积分系统
```

**经理遇到的问题 → 对应的功能**

```
"月底要手动统计"  → 统计分析 + 报表导出
"商品价格要调整"  → 商品管理 + 价格修改
```

**管理员遇到的问题 → 对应的功能**

```
"每个人能看什么要控制" → 用户管理 + 角色权限
```

**最终确定的六个模块，每一个都有来源，没有一个是"感觉应该有"加进去的：**

1. 用户与权限管理
2. 商品管理
3. 库存管理
4. 订单管理
5. 会员管理
6. 统计分析

> 💬 思考一下：如果去掉"统计分析"模块，经理每天会多做哪些额外的工作？

---

## 第四步：用户怎么使用这个系统

以店员的一天为例，走一遍最核心的使用流程：

```
店员登录系统
    ↓
顾客进店，问小米 14 还有没有货
    ↓
店员打开【库存管理】，搜索"小米14"，看到剩余库存
    ↓
顾客决定购买，店员打开【订单管理】→ 新建订单
    ↓
添加商品：小米14 × 1，选择支付方式：微信支付
    ↓
确认支付，系统自动：
    - 库存减 1
    - 生成订单记录
    - 如果顾客是会员，自动累积积分
    ↓
顾客问能不能办会员
    ↓
店员打开【会员管理】→ 注册新会员，填手机号和姓名
    ↓
完成，顾客离开
```

**这个流程告诉我们：订单、库存、会员三个模块是强关联的，不是独立的。**
下单要扣库存，支付要算积分，这些联动逻辑在设计时就要想清楚。

> 💬 思考一下：如果库存扣减失败了，但订单已经生成了，会出现什么问题？怎么解决？

---

## 第五步：第一版不做什么

需求很多，但第一版要克制。以下这些功能**暂时不做**，原因也写清楚：

| 暂时不做的功能 | 为什么先不做 |
|--------------|------------|
| 小程序商城 | 线下门店场景优先，线上是第二阶段 |
| 多门店管理 | 先把单店做稳，多店是扩展功能 |
| AI 销售分析 | 数据积累够了再做，现在没有训练数据 |
| 供应链管理 | 超出门店管理范围，是另一个系统 |
| Redis 缓存 | 第一版并发量不高，过度设计 |

**第一版的边界：一家门店、四种角色、六个核心模块、能跑通完整业务流程。**

---

## 总结：需求演变的完整路径

```
发现问题（运营数据散乱）
    ↓
确定用户（店员 / 经理 / 财务 / 管理员）
    ↓
按用户痛点推导功能（6个模块）
    ↓
想清楚核心使用流程（订单→库存→会员联动）
    ↓
划定第一版边界（不做什么）
    ↓
得到现在这个系统
```

**这就是需求演变的过程。**
你在这个项目里写的每一行代码，背后都有一个真实的原因。
