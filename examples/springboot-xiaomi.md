# 1. 项目概述

## 1.1 项目名称
小米之家门店管理系统（Mi Store Management System）

## 1.2 项目一句话定义
一个支持门店商品、库存、销售订单、会员和员工统一管理的后端系统。

## 1.3 项目类型
管理系统 / Web 应用后端

## 1.4 目标平台
桌面浏览器优先，PC 端使用

## 1.5 当前阶段
第一版 Demo 开发阶段

## 1.6 本次任务目标
先完成商品管理和库存管理的核心接口，再完成销售订单模块

# 2. 项目目标与成功标准

## 2.1 项目目标
完成一个可演示的门店管理系统最小可用版本，覆盖商品、库存、订单、会员四个核心模块

## 2.2 用户价值
- 目标用户是：门店经理、店员、财务人员
- 他们当前的主要问题是：商品、订单、库存数据分散，人工管理效率低、容易出错
- 这个项目提供的核心价值是：统一管理门店日常业务数据，提高运营效率

## 2.3 成功标准
- 可以管理商品（新增、编辑、上下架）
- 可以管理库存（入库、出库、库存预警）
- 可以创建和查询销售订单
- 可以管理会员信息和积分
- 接口无明显报错，数据流转符合预期

## 2.4 演示标准
- 使用假数据或 H2 内存数据库
- 可以演示商品录入 → 销售下单 → 库存扣减的主流程
- 接口返回结构清晰，便于课堂讲解

# 3. 用户角色定义

## 3.1 角色列表
- 角色 1：系统管理员（权限最高）
- 角色 2：门店经理（运营管理）
- 角色 3：店员（日常销售）
- 角色 4：财务人员（报表统计）

## 3.2 每个角色的权限定义

### 角色：系统管理员
**可以做的事：**
- 用户管理（新增、编辑、禁用）
- 角色权限配置
- 系统设置
- 数据维护

### 角色：门店经理
**可以做的事：**
- 查看销售报表
- 管理员工
- 审核库存
- 管理商品价格

**不可以做的事：**
- 修改系统配置
- 删除用户

### 角色：店员
**可以做的事：**
- 商品销售 / 开单
- 查询库存
- 会员登记

**不可以做的事：**
- 查看财务报表
- 修改商品价格

### 角色：财务人员
**可以做的事：**
- 销售统计
- 结算管理
- 财务报表查看

**不可以做的事：**
- 修改商品信息
- 管理用户

> 第一版可以用角色字段参数模拟权限，不强制接入 Spring Security。

# 4. 核心问题与使用场景

## 4.1 当前问题
- 商品库存靠人工 Excel 维护，容易出错
- 销售订单分散记录，无法统计汇总
- 会员信息管理混乱，积分无法自动计算

## 4.2 目标场景
- 店员销售时快速查询商品库存
- 创建订单后库存自动扣减
- 会员购买后自动累积积分

## 4.3 高频使用场景
- 查询商品列表和库存
- 创建销售订单
- 查询会员信息
- 查看销售统计报表

## 4.4 最关键主流程
店员查询商品库存
→ 创建销售订单（选择商品、数量、支付方式）
→ 订单支付完成，库存自动扣减
→ 会员积分自动累加
→ 门店经理查看当日销售报表

- 一个 Order 包含多个 OrderItem
- 每个 OrderItem 关联一个 Product
- Order 可选关联一个 Member（非会员也可下单）
- 订单支付完成后，自动扣减对应商品库存
- 订单支付完成后，若关联会员则自动累加积分

# 8. 业务规则

## 8.1 库存规则
- 下单时检查库存是否充足，不足则拒绝下单
- 订单支付完成后自动扣减库存
- 库存低于 min_stock 时，接口返回预警标记

## 8.2 积分规则
- 每消费 1 元累积 1 积分
- 积分在订单支付完成后自动计算并写入
- 会员等级根据累计积分自动判断：
  - 普通会员：0～999 分
  - 银卡会员：1000～4999 分
  - 金卡会员：5000～19999 分
  - 黑金会员：20000 分及以上

## 8.3 订单规则
- 订单创建后状态为「待支付」
- 调用支付接口后状态变为「已支付」
- 第一版不做真实支付，调用支付接口直接返回成功

## 8.4 数据规则
- 商品价格不能为负数
- 手机号格式需校验（11位数字）
- OrderItem 的 unit_price 在下单时快照，不随商品价格变动

# 10. 技术约束

## 10.1 技术栈要求
- 后端：Spring Boot + MySQL（或 H2 内存数据库）
- 接口格式：RESTful JSON
- 密码加密：BCrypt

## 10.2 实现原则
- 优先使用最简单稳定的实现
- 权限控制第一版用角色字段判断，不强制 Spring Security
- 非必要不引入额外依赖

## 10.3 复杂度限制
- 不引入 Redis 缓存
- 不做微服务拆分
- 不做 Docker 部署
- 不对接真实支付接口
- 不做 WebSocket 实时推送
