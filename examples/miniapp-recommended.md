# AI项目文档 — 签到打卡小程序

> 课程：微信小程序开发
> 案例类型：推荐项目
> 适用对象：AI 内部参照 + 学生查看示例

---

# 1. 项目概述

## 1.1 项目名称
签到打卡小程序

## 1.2 项目一句话定义
一个让老师发起签到、学生扫码完成打卡的微信小程序。

## 1.3 项目类型
小程序 / 工具类应用

## 1.4 目标平台
微信小程序（手机端）

## 1.5 当前阶段
第一版 Demo 开发阶段

## 1.6 本次任务目标
先完成老师发起签到和学生提交签到功能，再完成签到记录查询

---

# 2. 项目目标与成功标准

## 2.1 项目目标
完成一个可演示的签到流程，支持老师发起签到、学生打卡、老师查看记录

## 2.2 用户价值
- 目标用户是：老师和学生
- 他们当前的主要问题是：课堂点名靠叫名字，慢且容易漏人，没有记录
- 这个项目提供的核心价值是：线上签到，结果实时可查，有历史记录

## 2.3 成功标准
- 老师可以发起一次签到
- 学生可以输入签到码完成打卡
- 老师可以查看本次签到的已签到名单
- 防止重复签到

---

# 3. 用户角色定义

## 3.1 角色列表
- 角色 1：老师（发起签到、查看记录）
- 角色 2：学生（参与签到）

## 3.2 权限定义

### 角色：老师
- 发起签到（生成签到码）
- 结束签到
- 查看签到记录（谁签了、谁没签）

### 角色：学生
- 输入签到码完成打卡
- 查看自己的签到历史

---

# 4. 核心问题与使用场景

## 4.1 当前问题
- 叫名字点名浪费上课时间
- 纸质签到单容易丢失
- 不知道谁迟到了

## 4.2 目标场景
- 上课开始时老师发起签到，生成签到码
- 学生打开小程序输入签到码
- 老师查看签到结果

## 4.4 最关键主流程
老师点击「发起签到」→ 填写课程名称 → 生成签到码（如：1234）
→ 学生打开小程序 → 填写姓名和签到码 → 提交
→ 老师查看本次已签到学生列表

---

# 5. 功能范围

## 5.1 本期必须做
- 老师发起签到（生成4位签到码）
- 学生输入签到码 + 姓名完成打卡
- 老师查看本次签到名单
- 防止同一姓名重复签到

## 5.2 本期可选
- 老师手动结束签到
- 学生查看自己的历史签到记录

## 5.3 本期明确不做
- 不做微信登录（用姓名代替）
- 不做班级管理
- 不做考勤统计报表
- 不做消息推送
- 不做人脸识别

---

# 6. 页面与模块清单

## 6.1 页面列表
- 页面 1：首页（角色选择）
- 页面 2：老师发起签到页
- 页面 3：老师查看签到记录页
- 页面 4：学生签到页

## 6.2 每个页面的定义

### 页面：首页
**主要内容：** 两个入口按钮（我是老师 / 我是学生）

### 页面：老师发起签到页
**主要内容：** 课程名称输入框、发起签到按钮、签到码展示区
**主要操作：** 输入课程名 → 点击发起 → 显示签到码 → 查看签到人数

### 页面：学生签到页
**主要内容：** 姓名输入框、签到码输入框、提交按钮
**主要操作：** 填写姓名和签到码 → 提交 → 显示签到成功

# 7. 数据结构定义

## 7.1 实体列表
- CheckIn（签到活动）
- CheckInRecord（签到记录）

## 7.2 字段定义

### 实体：CheckIn（签到活动）

| 字段名 | 类型 | 含义 | 示例值 |
|---|---|---|---|
| _id | string | 文档ID | 自动生成 |
| courseName | string | 课程名称 | "Java基础" |
| code | string | 4位签到码 | "2847" |
| status | string | 状态 | "进行中" / "已结束" |
| createdAt | date | 发起时间 | 自动生成 |

### 实体：CheckInRecord（签到记录）

| 字段名 | 类型 | 含义 | 示例值 |
|---|---|---|---|
| _id | string | 文档ID | 自动生成 |
| checkInId | string | 关联签到活动ID | |
| studentName | string | 学生姓名 | "张三" |
| createdAt | date | 签到时间 | 自动生成 |

---

# 8. 业务规则

## 8.1 签到码规则
- 签到码为4位随机数字
- 每次发起签到生成新的签到码

## 8.2 防重复规则
- 同一签到活动中，相同姓名只能签到一次
- 重复提交时提示「你已经签到过了」

## 8.3 数据规则
- 姓名和签到码不能为空
- 签到码不正确时提示错误

---

# 9. 界面与交互要求

## 9.1 整体风格
简洁清晰，适合手机端操作，按钮足够大

## 9.2 反馈要求
- 签到成功显示成功提示页面
- 签到码错误显示错误提示
- 重复签到显示提示

---

# 10. 技术约束

## 10.1 技术栈要求
- 微信小程序原生开发（WXML + WXSS + JS）
- 数据存储使用微信云开发（云数据库）

## 10.2 实现原则
- 优先使用云开发，不搭建独立后端
- 签到码用随机数生成，不需要复杂算法

## 10.3 复杂度限制
- 不做微信登录
- 不做云函数（直接用数据库 API）
- 不做图片存储

---

## 11.2 禁止事项
- 不做微信登录
