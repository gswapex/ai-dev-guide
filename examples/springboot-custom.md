# AI项目文档 — 宿舍报修管理系统

> 课程：Spring Boot / Java 后端
> 案例类型：自定义项目（多角色场景）
> 适用对象：AI 内部参照 + 学生查看示例

# 1. 项目概述

## 1.1 项目名称
宿舍报修管理系统

## 1.2 项目一句话定义
一个让学生提交宿舍报修申请、后勤人员处理工单的管理系统。

## 1.3 项目类型
管理系统 / Web 应用

## 1.4 目标平台
桌面浏览器优先

## 1.5 当前阶段
第一版 Demo 开发阶段

## 1.6 本次任务目标
先完成报修工单的提交和状态流转接口

---

# 2. 项目目标与成功标准

## 2.1 项目目标
完成一个可演示的报修流程，支持学生提交工单、后勤人员处理和关闭工单

## 2.2 用户价值
- 目标用户是：在校学生和后勤维修人员
- 他们当前的主要问题是：报修靠口头或微信，处理进度不透明，容易遗漏
- 这个项目提供的核心价值是：线上化报修流程，状态实时可查

## 2.3 成功标准
- 学生可以提交报修申请
- 后勤人员可以查看待处理工单并更新状态
- 工单状态流转正确（待处理 → 处理中 → 已完成）

## 2.4 演示标准
- 可以演示完整的报修 → 处理 → 完成流程
- 使用假数据即可

# 3. 用户角色定义

## 3.1 角色列表
- 角色 1：学生（提交报修）
- 角色 2：后勤人员（处理工单）

## 3.2 每个角色的权限定义

### 角色：学生
**可以做的事：**
- 提交报修申请
- 查看自己提交的工单状态

**不可以做的事：**
- 查看其他学生的工单
- 修改工单状态

### 角色：后勤人员
**可以做的事：**
- 查看所有待处理工单
- 更新工单状态（接单、完成）
- 填写处理备注

**不可以做的事：**
- 删除工单记录

---

# 4. 核心问题与使用场景

## 4.1 当前问题
- 报修靠微信群喊，容易被刷屏淹没
- 后勤不知道哪些工单还没处理
- 学生不知道自己的报修进度

## 4.2 目标场景
- 学生发现宿舍设施损坏，提交报修单
- 后勤人员查看待处理工单，接单处理
- 处理完成后更新状态，学生可查看结果

## 4.3 高频使用场景
- 学生提交报修
- 后勤查看工单列表
- 后勤更新工单状态

## 4.4 最关键主流程
学生提交报修申请（填写宿舍号、故障描述）
→ 工单状态变为「待处理」
→ 后勤人员查看工单列表
→ 后勤接单，状态变为「处理中」
→ 后勤完成维修，状态变为「已完成」
→ 学生查询工单状态确认完成

# 7. 数据结构定义

## 7.1 实体列表
- RepairOrder（报修工单）

## 7.2 每个实体的字段定义

### 实体：RepairOrder

| 字段名 | 类型 | 是否必填 | 含义 | 示例值 |
|---|---|---|---|---|
| id | Long | 是 | 主键自增 | 1 |
| dormNo | String | 是 | 宿舍号 | "A栋302" |
| faultType | String | 是 | 故障类型 | "水电" / "家具" / "其他" |
| description | String | 是 | 故障描述 | "水龙头漏水" |
| status | String | 是 | 工单状态 | "待处理" / "处理中" / "已完成" |
| remark | String | 否 | 后勤备注 | "已更换零件" |
| createdAt | DateTime | 是 | 提交时间 | "2024-03-10 10:00" |
| updatedAt | DateTime | 是 | 最后更新时间 | "2024-03-10 14:00" |

## 7.3 实体关系说明
- 第一版只有 RepairOrder 一个核心实体
- 不做用户表，用 dormNo 标识提交人

# 8. 业务规则

## 8.1 状态规则
- 工单状态只允许为：待处理、处理中、已完成
- 状态流转方向：待处理 → 处理中 → 已完成
- 不允许状态回退

## 8.2 权限规则
- 第一版用请求参数 role 区分角色（student / staff）
- 不做真实登录验证

## 8.3 数据规则
- 宿舍号和故障描述不能为空
- 状态修改后自动更新 updatedAt

## 8.4 操作规则
- 提交成功返回工单 ID
- 状态更新成功返回最新工单信息

# 10. 技术约束

## 10.1 技术栈要求
- Spring Boot + MySQL（或 H2 内存数据库）
- RESTful JSON 接口

## 10.2 实现原则
- 状态流转用简单字符串字段，不做状态机框架
- 权限用参数模拟，不做 Spring Security

## 10.3 复杂度限制
- 不引入消息队列
- 不做文件上传
- 不做复杂权限框架
